# 子代理驱动开发

通过将任务委派给子代理、审查其产出并迭代来实现计划。

## 流程

### 1. 分解计划
- 将规格/计划拆分为离散任务
- 每个任务应：可独立实现、可测试、可审查
- 识别任务间依赖（A 必须在 B 之前完成）
- 按依赖排序，再按风险排序（更难/风险更高的先做）

### 2. 逐个实现
对每个任务：

1. **向子代理简报**：给任务描述、相关规格章节、上下文
2. **要求 TDD**：子代理必须先写测试，再写实现
3. **审查产出**：
   - 测试通过了吗？
   - 实现符合规格吗？
   - 有明显问题吗（安全、性能、风格）？
4. **迭代**：发现问题则反馈给子代理修复
5. **接受**：测试通过且审查干净，进入下一个任务

### 3. 集成检查
所有任务完成后：
- 运行完整测试套件
- 检查集成问题（单独工作但一起就坏的任务）
- 验证规格中的所有验收标准都已满足

### 4. 最终审查
- 审查完整 diff
- 检查任务边界的一致性
- 确保没有死代码或调试残留

## 规则
- 不允许子代理跳过 TDD
- 总是审查子代理产出 — 不盲目信任
- 子代理卡住 3 次迭代后，自己接手该任务
- 保持任务小 — 子代理应在 15 分钟内完成一个
- 任务跨多文件边界时，进一步拆分
- 在提交信息中记录与计划的任何偏差

## 子代理简报模板
```
任务：<一行描述>
上下文：<相关规格章节>
修改文件：<列表>
方法：<提示，或"照常 TDD">
完成标准：<测试通过，验收标准满足>
```
