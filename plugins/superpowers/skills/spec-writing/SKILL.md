# 规格编写

将模糊的想法转化为初级工程师也能实现的清晰规格。

## 结构

### 1. 问题陈述
一段话。今天存在什么痛点？谁感受到了？如果不做会怎样？

### 2. 方案概述
一段话。我们在做什么？它如何解决痛点？

### 3. 用户故事
```
作为一个 <角色>，
我想要 <能力>，
以便 <目的>。
```
写 3-7 个故事，每个必须可独立测试。

### 4. 验收标准
每个用户故事列 2-5 条标准，每条必须可验证：
- "给定 X，当 Y，那么 Z"
- 不是"系统应该快" — "95 分位响应时间 < 200ms"

### 5. 技术设计
- 数据模型（schema、关系）
- API 端点（方法、路径、请求、响应）
- 关键算法或流程
- 外部依赖
- 迁移/上线计划（如修改现有系统）

### 6. 不在范围内
明确列出本规格不涵盖的内容，防止范围蔓延。

### 7. 待解决问题
列出实现前或实现中需要回答的未解决问题。

## 规则
- 分段展示给用户（不要一次性全给）
- 每段足够短，可以读完消化
- 等用户确认后再进入下一段
- 如果用户说"直接做"，仍然写最小规格 — 后面能省时间
- 规格要具体到任何工程师都能无需进一步提问就实现
