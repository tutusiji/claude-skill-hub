# 测试驱动开发

严格的红-绿-重构 TDD。没有失败测试就不写生产代码。

## 循环

### 红 — 写失败测试
1. 为一个行为写一个测试
2. 运行它，必须失败（编译错误或断言失败）
3. 如果通过了，说明测试写错了 — 重写

### 绿 — 让它通过
1. 写最少量的代码让测试通过
2. 不添加测试不要求的的任何东西
3. 运行测试，必须通过

### 重构 — 清理
1. 在不改变行为的前提下提升代码质量
2. 每次重构步骤后运行测试
3. 如果任何测试失败，撤销重试

## 规则

- 永远不在测试之前写生产代码
- 永远不写超过"让它失败"所需的测试
- 永远不写超过"让它通过"所需的生产代码
- 一个测试一个行为 — 如果测了两件事，拆成两个测试
- 测试名描述行为：`should_return_empty_list_when_no_results` 而非 `test1`
- Mock 外部依赖，不 mock 自己的代码
- 发现 bug 时，先写复现测试，再修复

## 应避免的反模式
- 提前写所有测试（写一个、过一个、循环）
- 测试实现细节而非行为
- 跳过重构步骤
- 代码写完后补测试"凑覆盖率"
