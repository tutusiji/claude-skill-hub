# 发布

以工程规范发布当前分支。

## 流程

1. 验证已运行 review（未运行则警告）
2. 检查工作树干净（无未提交变更）
3. 如需 squash 或整理提交
4. 从 diff 生成 PR 标题和描述
5. 推送分支并创建 PR
6. 报告 PR URL

## 提交卫生

- 每个提交应该是一个逻辑工作单元
- 提交信息遵循 Conventional Commits 格式
- 最终 PR 中不出现 "wip" 或 "fix" 提交
- 将琐碎的 fixup 提交合并到父提交

## PR 描述模板

```
## 做了什么
<一段话：这个 PR 做了什么以及为什么>

## 变更
- <关键变更列表>

## 测试
- <如何测试>
- <测试了什么>

## 审查备注
- <审查者需要注意的事项>
```

## 规则
- 不 force-push 到 main
- 不未经 review 就发布（警告但不阻止）
- CI 红时不发布 — 先修复
- 如有 schema 变更，附带迁移步骤
