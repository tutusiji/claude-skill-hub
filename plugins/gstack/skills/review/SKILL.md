# 代码审查

以生产级标准审查当前分支相对 main 的变更。

## 流程

1. `git diff main...HEAD` — 获取完整 diff
2. 按严重级别分类发现：**阻断 / 警告 / 建议**
3. 系统检查以下每个类别

## 审查类别

### 安全性
- 硬编码密钥、令牌、API key
- SQL 注入、XSS、CSRF 漏洞
- 不安全反序列化
- 外部接口缺少输入校验
- 文件访问权限过于宽松

### 正确性
- 差一错误、空指针解引用
- 并发代码中的竞态条件
- I/O 操作缺失错误处理
- 类型转换错误
- 边界情况：空输入、最大值、Unicode

### 性能
- N+1 查询
- 无界循环或递归
- 缺失数据库索引
- 热路径中的大内存分配
- 异步上下文中的同步 I/O

### 设计
- 范围蔓延 — 与功能无关的变更
- 缺失或不正确的抽象
- 公共 API 的破坏性变更
- 与代码库约定不一致的命名或模式

## 输出格式

```
## 审查报告：<分支名>

### 阻断（合并前必须修复）
- [文件:行号] 描述

### 警告（应修复，不阻断）
- [文件:行号] 描述

### 建议（可选优化）
- [文件:行号] 描述

### 汇总
N 个阻断，M 个警告，K 个建议。<一句话结论>
```

## 规则
- 读实际 diff，不只看文件名
- 通过阅读周围代码验证发现
- 不提与现有约定矛盾的风格修改
- 如果 diff 干净，直说 — 不要硬找问题
