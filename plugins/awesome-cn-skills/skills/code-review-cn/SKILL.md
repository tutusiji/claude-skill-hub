# 中文代码审查

按国内团队工程规范审查代码变更，输出中文报告。

## 审查清单

### 命名规范
- 变量/函数名使用有意义的中文拼音或英文，禁止 a/b/tmp
- 常量全大写下划线: `MAX_RETRY_COUNT`
- 类名大驼峰: `UserService`
- 函数名小驼峰或下划线，与项目现有风格一致
- 接口名加 `I` 前缀或按项目约定

### 注释规范
- 公共函数必须有文档注释
- 复杂逻辑必须有行内注释说明原因
- 注释用中文，描述"为什么"而非"做了什么"
- TODO 注释格式: `// TODO(姓名): 描述`

### 异常处理
- 不允许空 catch 块
- 异常必须记录日志
- 业务异常和系统异常分开处理
- 错误信息对用户友好，不暴露技术细节

### 日志规范
- 关键操作记录 INFO 日志
- 异常记录 ERROR 日志，含完整堆栈
- 敏感信息脱敏（手机号、身份证、密码）
- 日志格式统一，含时间、级别、模块、traceId

### 安全检查
- SQL 参数化查询，禁止字符串拼接
- 用户输入必须校验和转义
- 文件上传检查类型和大小
- 接口鉴权检查

## 输出格式

```
## 代码审查报告

### 阻断问题（必须修复）
- [文件:行号] 问题描述 → 修复建议

### 警告（建议修复）
- [文件:行号] 问题描述

### 建议（可选优化）
- [文件:行号] 优化建议

### 汇总
阻断 N 个，警告 M 个，建议 K 个
结论: <可合并 / 修复后合并 / 需重大修改>
```
